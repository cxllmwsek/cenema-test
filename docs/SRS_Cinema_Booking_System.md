# Software Requirements Specification (SRS)
## ระบบจองตั๋วโรงภาพยนตร์ (Cinema Booking System)

**เวอร์ชัน:** 1.0
**วันที่:** 13 กรกฎาคม 2569
**ภาษาที่ใช้พัฒนา:** Python 3.x
**ทีมพัฒนา:** 5 คน

---

## 1. บทนำ (Introduction)

### 1.1 วัตถุประสงค์ (Purpose)
เอกสารนี้ระบุข้อกำหนดของระบบจองตั๋วโรงภาพยนตร์ เพื่อใช้เป็นแนวทางในการออกแบบและพัฒนาโปรแกรมโดยทีมพัฒนา 5 คน โดยเน้นการออกแบบเชิงวัตถุ (Object-Oriented Programming) เพื่อใช้เป็นผลงานพอร์ตโฟลิโอบน GitHub

### 1.2 ขอบเขตของระบบ (Scope)
ระบบนี้เป็นโปรแกรมจำลองการจองตั๋วภาพยนตร์ ทำงานผ่าน Command Line Interface (CLI) โดยรองรับ:
- การจัดการผู้ใช้งาน (สมาชิก/ผู้ดูแลระบบ)
- การจัดการภาพยนตร์และรอบฉาย
- การเลือกและจองที่นั่ง
- การชำระเงินหลายรูปแบบ
- การบันทึกข้อมูลถาวรและออกรายงาน

ระบบนี้ **ไม่รวม** ส่วนของ GUI/Web Interface, ระบบชำระเงินจริงผ่านธนาคาร, และการเชื่อมต่อ API ภายนอก (อยู่นอกขอบเขต เว้นแต่ทีมต้องการขยายเพิ่มในอนาคต)

### 1.3 นิยามและคำย่อ (Definitions)
| คำศัพท์ | ความหมาย |
|---|---|
| Member | สมาชิกที่สมัครใช้งานระบบ |
| Admin | ผู้ดูแลระบบ จัดการข้อมูลหนังและรอบฉาย |
| Showtime | รอบฉายภาพยนตร์ในแต่ละวัน/เวลา |
| Booking | รายการจองที่นั่งของผู้ใช้ |
| Seat Map | ผังที่นั่งในโรงภาพยนตร์ |

### 1.4 ผู้ใช้งานระบบ (User Types)
1. **Guest** — ดูรอบหนังได้ แต่ต้องสมัครสมาชิกก่อนจอง
2. **Member** — จองตั๋ว, ชำระเงิน, ดูประวัติการจอง, สะสมแต้ม
3. **Admin** — จัดการหนัง/รอบฉาย/โรง, ดูรายงานยอดขาย

## 2. ภาพรวมระบบ (Overall Description)

### 2.1 สถาปัตยกรรมโดยรวม
ระบบแบ่งออกเป็น 5 โมดูลหลัก สอดคล้องกับการแบ่งงานทีม 5 คน:

| โมดูล | ผู้รับผิดชอบ | คลาสหลัก |
|---|---|---|
| M1: User Management | คนที่ 1 | `User`, `Member`, `Admin` |
| M2: Movie & Showtime | คนที่ 2 | `Movie`, `Showtime`, `Theater` |
| M3: Seat & Booking | คนที่ 3 | `Seat`, `SeatMap`, `Booking` |
| M4: Payment | คนที่ 4 | `Payment` (abstract), `CreditCardPayment`, `CashPayment`, `Invoice` |
| M5: Data & Reporting | คนที่ 5 | `DataStore`, `Report`, persistence layer (JSON/SQLite) |

`main.py` เป็นจุดรวมที่เรียกใช้ทุกโมดูลเข้าด้วยกัน

### 2.2 ข้อจำกัดในการออกแบบ (Design Constraints)
- ต้องเขียนด้วยหลักการ OOP ครบ 4 เสาหลัก: Encapsulation, Inheritance, Polymorphism, Abstraction
- ต้องใช้ Composition ในการเชื่อมโยง class เช่น `Booking` มี `Seat`, `Movie`, `Member`
- ทุก class ต้องมี docstring และ type hint
- ห้ามเข้าถึงหรือแก้ไข attribute แบบ private จากภายนอก class โดยตรง ต้องผ่าน method/property เท่านั้น

## 3. ข้อกำหนดเชิงฟังก์ชัน (Functional Requirements)

### FR-1: การจัดการผู้ใช้ (M1)
- FR-1.1: ผู้ใช้ต้องสามารถสมัครสมาชิก (register) ด้วยชื่อ, อีเมล, รหัสผ่าน
- FR-1.2: ผู้ใช้ต้องสามารถเข้าสู่ระบบ (login) และออกจากระบบ (logout)
- FR-1.3: Member ต้องมีระบบสะสมแต้ม (points) จากการจอง
- FR-1.4: Admin ต้องสามารถเพิ่ม/ลบ/แก้ไขข้อมูล Movie และ Showtime ได้ (สิทธิ์ที่ Member ไม่มี)
- FR-1.5: ระบบต้องตรวจสอบสิทธิ์ (authorization) ก่อนอนุญาตให้ทำ action ที่จำกัดเฉพาะ Admin

### FR-2: การจัดการภาพยนตร์และรอบฉาย (M2)
- FR-2.1: Admin สามารถเพิ่มภาพยนตร์ใหม่ (ชื่อ, ประเภท, ความยาว, เรตอายุ)
- FR-2.2: Admin สามารถกำหนดรอบฉาย (วันที่, เวลา, โรงภาพยนตร์) ให้กับภาพยนตร์
- FR-2.3: ผู้ใช้ต้องสามารถค้นหาภาพยนตร์ตามชื่อ/ประเภท/วันที่ฉายได้
- FR-2.4: ระบบต้องแสดงรอบฉายที่ยังมีที่นั่งว่างเท่านั้น (หรือระบุสถานะเต็ม)

### FR-3: การเลือกที่นั่งและจอง (M3)
- FR-3.1: ระบบต้องแสดงผังที่นั่ง (Seat Map) ของแต่ละรอบฉาย พร้อมสถานะ (ว่าง/จองแล้ว)
- FR-3.2: ผู้ใช้ต้องสามารถเลือกที่นั่งได้มากกว่า 1 ที่นั่งต่อการจอง 1 ครั้ง
- FR-3.3: ระบบต้องป้องกันการจองที่นั่งซ้ำ (race condition ในกรณีจองพร้อมกัน — อย่างน้อยต้อง validate ก่อนยืนยัน)
- FR-3.4: ผู้ใช้ต้องสามารถยกเลิกการจองได้ก่อนถึงเวลาที่กำหนด (เช่น ก่อนฉาย 30 นาที)
- FR-3.5: ระบบต้องเก็บประวัติการจองของแต่ละ Member

### FR-4: การชำระเงิน (M4)
- FR-4.1: ระบบต้องรองรับการชำระเงินอย่างน้อย 2 รูปแบบ (เช่น บัตรเครดิต, เงินสด/wallet จำลอง)
- FR-4.2: ระบบต้องคำนวณส่วนลด (ถ้ามี) เช่น ส่วนลดสมาชิก, โปรโมชั่น
- FR-4.3: เมื่อชำระเงินสำเร็จ ระบบต้องออกใบเสร็จ (Invoice) พร้อมเลขที่อ้างอิง
- FR-4.4: หากชำระเงินไม่สำเร็จ ที่นั่งที่เลือกไว้ต้องถูกปลดล็อกกลับเป็นสถานะว่าง

### FR-5: การจัดเก็บข้อมูลและรายงาน (M5)
- FR-5.1: ข้อมูลทั้งหมด (ผู้ใช้, หนัง, รอบฉาย, การจอง) ต้องถูกบันทึกแบบถาวร (persist) ด้วย JSON หรือ SQLite
- FR-5.2: ระบบต้องโหลดข้อมูลจากไฟล์/ฐานข้อมูลได้ถูกต้องเมื่อเปิดโปรแกรมใหม่
- FR-5.3: Admin ต้องสามารถดูรายงานยอดขายรายวัน/รายเรื่อง/รายรอบฉายได้
- FR-5.4: ระบบต้องมี error handling เมื่อไฟล์ข้อมูลเสียหายหรือหาไม่พบ

## 4. ข้อกำหนดเชิงไม่ใช่ฟังก์ชัน (Non-Functional Requirements)

| หมวด | ข้อกำหนด |
|---|---|
| Maintainability | โค้ดต้องแบ่ง module ชัดเจนตาม class responsibility เดียว (Single Responsibility) |
| Readability | ทุกฟังก์ชัน/คลาสต้องมี docstring อธิบายหน้าที่ |
| Testability | แต่ละโมดูลต้องมี unit test (pytest) ครอบคลุมอย่างน้อย happy path + 1 edge case |
| Extensibility | การเพิ่มวิธีชำระเงินใหม่ หรือประเภทผู้ใช้ใหม่ ต้องทำได้โดยไม่แก้โค้ดเดิม (Open-Closed Principle) |
| Version Control | ใช้ Git branching (feature branch ต่อคน) และ Pull Request ก่อน merge เข้า main |
| Documentation | ต้องมี README.md อธิบายวิธีติดตั้ง/รัน และ class diagram ประกอบ |

## 8. เกณฑ์การยอมรับ (Acceptance Criteria)
- ผู้ใช้สามารถจองตั๋วครบวงจรตั้งแต่ค้นหาหนังจนถึงได้ใบเสร็จ โดยไม่มี error
- โค้ดใช้ OOP ครบ 4 เสาหลักอย่างมีเหตุผล (ไม่ยัดเยียด)
- มี unit test ผ่านอย่างน้อย 80% ของฟังก์ชันหลัก
- มี README ที่อธิบายวิธีติดตั้งและรันโปรเจ็คได้ชัดเจน
- Git history แสดงการทำงานร่วมกันของทั้ง 5 คนอย่างชัดเจน (commit/PR)
